# Часть 1: память и связность

## 1.1 Формальная граница

Точная цитируемая граница, без размывания:

**Refcounting не собирает циклы — это свойство метода, а не реализации.** Bacon & Rajan ("Concurrent Cycle Collection in the Refcounted Sea of Objects", 2001) формализуют: любой чисто локальный (per-object) refcount инвариант "count == число входящих ссылок" сохраняется при декременте до нуля, но для группы объектов, взаимно ссылающихся по циклу, декремент внешней ссылки оставляет внутренние счётчики > 0 при нуле реально достижимых внешних корней. Единственный практичный обход без полной трассировки — **trial deletion** (пробное удаление): гипотетически вычесть внутренние ссылки в подграфе-кандидате и проверить, упал ли счётчик до нуля. Но trial deletion — это уже частичная трассировка подграфа: она требует обхода, то есть паузы и знания структуры графа в рантайме.

Отсюда триада, недостижимая **одновременно** в общем случае:

```
(A) произвольно связный ИЗМЕНЯЕМЫЙ граф (циклы вводятся динамически)
(B) автоматическое освобождение без паузы трассировки
(C) без рантайма (без метаданных обхода, без cycle collector thread)
```

Любые две — можно. Все три — нет. GC-языки берут (A)+(B) ценой (C). Rust берёт (B)+(C) ценой ограничения (A) (borrow checker + явный `Weak`). MLC сейчас декларирует (B)+(C) и **честно ограничивает (A)** — цитата из `MEMORY_MODEL.md`: *"Циклические структуры требуют `Weak<T>`. Забытый `Weak` → утечка. Компилятор не детектирует."* Это математически корректная позиция, но эргономически непроверенная — на что прямо указывает `grep "Weak<"` = 0.

## 1.2 Цена эргономики по каждой системе

| Система | Что даёт | Чем платит | Применимо к MLC (без GC/рантайма)? |
|---|---|---|---|
| Pony (`iso`/`val`/`ref`/`box`/`tag`) | Безопасный изменяемый граф в конкуренции без общего GC на уровне capabilities | Per-actor GC-рантайм всё же есть (mark-sweep внутри актора); reference capabilities — нагрузка на систему типов и на программиста (5 капабилити на каждую ссылку) | Частично: капабилити как аннотации — да; освобождение всё равно требует акторного GC — **нет** |
| Vale (generational references) | Изменяемый alias без borrow-checker трения, use-after-free ловится | Рантайм-накладные: generation-номер в каждом аллокейте + проверка поколения при разыменовании (не ноль!); ещё исследовательский | Нарушает "нулевой runtime overhead" — **нет как дефолт**, возможно как opt-in unsafe |
| Verona (регионы) | Регион = единица владения и освобождения; внутри региона — свободный изменяемый граф | Рантайм регионов + ограничение: между регионами только `iso`-переходы; исследовательский | Идея регионов — **да** (см. 1.4); рантайм-часть — нет |
| Koka/Perceus | Снижение RC-трафика через reuse-анализ (FBIP), decrement/увеличение элиминируются статически | Не решает циклы вообще; требует precise-refcount дисциплины в IR | Reuse-анализ применим как оптимизация codegen (см. Часть 3), но **не расширяет** выразительность графа |
| Rust `Rc<RefCell>`+`Weak` | (B)+(C) честно | Программист вручную разрывает циклы `Weak`; `RefCell` двигает borrow-проверку в рантайм (panic) | **Прямой аналог текущего MLC** — это и есть потолок без GC |
| Swift ARC | Прозрачный RC | `weak`/`unowned` вручную для циклов (та же граница); ARC-трафик атомарен | Прямой аналог MLC COW+Shared |

Вывод: **ни одна система не даёт "триаду" бесплатно.** Все, кто даёт (A)+(B), приносят рантайм (Pony, Verona, Vale-generations). Значит MLC при своих ограничениях уже стоит на правильной стороне границы, и вопрос — только эргономика сахара вокруг `Weak`, не расширение выразительности.

## 1.3 Три уровня рекомендаций

### Уровень 0 — дёшево, только сахар над `Shared`/`Owned`/`Weak`

Проблема из текста: `Weak<T>` документирован, но 0 использований → эргономика не обкатана. Причина гипотетически (требует прверки на реальном графе) в том, что нет удобного паттерна "создать Shared → получить Weak → апгрейднуть Weak обратно". Предлагаемый сахар, весь транслируется в `std::shared_ptr`/`std::weak_ptr`:

```mlc
// downgrade: Shared<T> -> Weak<T>   →  std::weak_ptr(sp)
let parent_ref = parent.weak()

// upgrade: Weak<T> -> Option<Shared<T>>  →  wp.lock() + проверка на nullptr
match child.parent.upgrade() {
  Some(p) => use(p),
  None    => handle_dead_parent()
}
```

`.weak()` и `.upgrade()` — методы в runtime на `Shared`/`Weak`, ноль нового рантайма (это `std::weak_ptr::lock()`). `.upgrade()` возвращающий `Option` — единственно корректная сигнатура (объект мог умереть), что автоматически совместимо с существующим `match`. Класс миграции: **non-breaking sugar**. Реализуется через `extend Weak<T>` в runtime-заголовке + правило в `builtin_method_return_type` checker.

### Уровень 1 — облегчённый статический анализ (НЕ borrow checker)

Не полный alias-анализ. Узкая цель: **предупреждать о потенциальном цикле** там, где тип рекурсивно ссылается на себя через `Shared` без `Weak` на обратной стороне. Алгоритм (один проход по `TypeRegistry`, данные уже есть — см. `AdtIndex.constructor_parameters`):

```
1. Построить граф достижимости типов по полям: ребро A→B, если поле A имеет
   тип Shared<B> (или [Shared<B>], или Shared<B> внутри generic).
2. Найти SCC (сильно связные компоненты) — стандартный Tarjan, O(V+E).
3. Для каждого SCC размера > 1 (или самопетля): если НИ одно ребро цикла
   не идёт через Weak<_> — выдать warning W-CYCLE с span объявления типа.
```

Это диагностика уровня lint (вписывается в план G2 — "синтаксис + ограниченная семантика"), не блокирующая ошибка. Синтаксис не меняется — только новая диагностика:

```
warning[W-CYCLE]: recursive type cycle Parent -> Child -> Parent through Shared
  without Weak on the back edge; may leak. Consider Weak<Parent> on Child.parent.
  --> node.mlc:3:1
```

Пайплайн: реализуется целиком внутри `checker/` как отдельный проход после `build_registry`, читает `adt_index`. Класс миграции: **non-breaking** (новый warning). Стоимость: ~один файл `checker/cycle_lint.mlc`, Tarjan на существующих `Map`. Важно: это *эвристика достижимости по типам*, ложные срабатывания возможны (не каждый структурный цикл типов = рантайм-цикл значений), поэтому именно warning с suppression, а не error.

### Уровень 2 — недостижимо без нарушения "без GC/рантайма"

Явно, без иллюзий:

- **Автоматический сбор динамически образованных циклов** `Shared<->Shared` без `Weak`. Требует cycle collector (trial deletion = частичная трассировка = рантайм-пауза + метаданные). Нельзя. Остаётся ручной `Weak` или утечка.
- **Прозрачный изменяемый alias одного объекта из N мест** (JS/Ruby-семантика `a = b; b.push(x)` видно в `a`). MLC отвечает на это COW — но COW даёт *value*-семантику, не *reference*-семантику. Настоящий shared-mutable требует `Shared<Mutable<T>>` c явным API (как `Rc<RefCell>`), с рантайм-проверкой borrow или без гарантий. Прозрачно — нельзя.
- **Обнаружение утечки цикла в общем случае** (не эвристика по типам, а по значениям в рантайме) — только трассировка. Нельзя.

## 1.4 Регион/арена как реалистичный выигрыш

Самый честный кандидат на реальную пользу для графа, живущего в пределах одной функции: **scoped bump-arena, ноль refcounting внутри, единое освобождение на выходе.** Это ровно случай AST-узлов внутри одного прохода компилятора — а `ast.mlc` показывает массовое `Shared<Expr>`/`Shared<Decl>`, где каждый узел несёт атомарный refcount, которого при построении дерева в одной функции не нужно.

Предлагаемый синтаксис (гипотеза, требует прототипа — но с ясной C++-трансляцией):

```mlc
region r do
  // Внутри r: аллокации через r ссылаются на bump-буфер r.
  // Ссылки — region-local, refcount НЕ инкрементится.
  let root = r.alloc(Node { children: [], value: 1 })
  let leaf = r.alloc(Node { children: [], value: 2 })
  // циклы ВНУТРИ региона допустимы: освобождение — не по refcount, а по концу scope
  build_tree(root, leaf)
  serialize(root)      // результат должен НЕ содержать region-ссылок наружу
end   // весь буфер r освобождается разом, один free
```

C++-трансляция: `region` → RAII-обёртка над `std::pmr::monotonic_buffer_resource` (C++17, в C++20 стабильно), `r.alloc(...)` → placement в этот resource. Единое освобождение = деструктор буфера. Ноль per-node refcount внутри.

**Математическое ограничение, которое обязан ловить checker (иначе UB):** ни одна ссылка на объект региона не должна пережить `end` региона. Это узкий escape-анализ (см. Часть 1b), НЕ общий borrow checker: региональные значения помечены типом (например `Region<'r, Node>` фантомным параметром `'r` — MLC уже планирует phantom types в C1), и правило — `Region<'r, _>` не может появиться в типе возврата/поля, живущего дольше `'r`. Класс миграции: **нужен отдельный прототип** (новый синтаксис + escape-правило + pmr в runtime). Это самая дорогая, но и единственная рекомендация, реально расширяющая выразительность графов без нарушения "без GC".

# Часть 1b: замыкания — сколько динамики без рантайма

## 1b.1 Точная граница

| Класс замыкания | Время жизни | C++ форма | Аллокация / RC |
|---|---|---|---|
| Non-escaping | не переживает вызов/scope, где создан | inline лямбда / template-параметр `template<class F>` (как Rust `impl Fn`) | ноль |
| Escaping, моно-тип | переживает, но конкретный тип известен | лямбда, хранимая в `auto`-поле шаблонной структуры | ноль RC, но тип "заражает" контейнер |
| Escaping, гетерогенно | сохранена в поле разнородной коллекции / за границей потока / рекурсивно | `std::function` / `std::move_only_function` (C++23) / `shared_ptr<...>` | **type erasure: vtable + heap** |

Граница формальна: если множество возможных конкретных типов замыкания, попадающих в одну ячейку хранения, **статически не единично** (гетерогенный `[SomeFn]`, поле с разными лямбдами в разных путях, cross-thread), компилятор обязан стереть тип — это тот же потолок, ради которого в C++ есть `std::function`, в Rust — `Box<dyn Fn>`. **Уровень 2 для замыканий:** гетерогенно-хранимое или cross-thread замыкание математически требует type erasure. Убрать нельзя.

## 1b.2 Swift `@escaping`/non-escaping как проверенный образец

Swift делает **дефолтом non-escaping** для параметров-замыканий: `func f(_ body: () -> Void)` — `body` не может пережить `f`, компилятор это проверяет, и такое замыкание не требует heap-аллокации/ARC-retain захваченного. Чтобы замыкание пережило вызов (сохранить в поле, передать в `DispatchQueue.async`), нужен явный маркер `@escaping`. Это **облегчённый** анализ (не Rust-borrow): проверяется ровно один битовый факт на параметр — "может ли значение утечь за пределы вызова" — через простые правила (сохранение в свойство, передача в другой escaping-параметр, возврат). Никакого lifetime-lattice, никаких регионов. Именно эта узость делает его практичным и годами обкатанным.

## 1b.3 Стоит ли MLC инвертировать дефолт

Прямая привязка к тексту: `MEMORY_MODEL.md` — *"Генерируется C++ lambda с `[=]` capture"*, а `mlc2.md/G1` — *"при непустом списке захватываемых имён префикс `[&]`"*. То есть текущая модель: захват по значению COW-контейнеров + boxing там, где замыкание где-то хранится. Задача прямо называет это *"причиной исходного кризиса скорости C++ сборки: повсеместные `shared_ptr<T>`/`std::function<...>`"*.

**Рекомендация: да, инвертировать — сделать дефолтом non-escaping (template + inline-лямбда), escaping — выводимым, с явным маркером только для спорных случаев.** Правило вывода (узкий escape-анализ, НЕ alias-анализ):

```
Замыкание ESCAPES, если хотя бы одно:
  (1) присвоено в поле record/сохранено в let, переживающий текущую функцию;
  (2) возвращено из функции (return type содержит Fn(...));
  (3) передано в параметр, помеченный escaping;
  (4) захвачено внутри Spawn (cross-thread, ExprSpawn в ast.mlc).
Иначе — non-escaping.
```

Синтаксис:
- Параметр-замыкание по умолчанию non-escaping → codegen как `template<class F>` (ноль RC).
- Escaping-параметр помечается в типе: `handler: escaping Fn(Request) -> Response` → codegen как `std::move_only_function` (C++23, пометить как >C++20) или `std::function` (C++20).
- Возврат замыкания (`-> (T -> U)`) автоматически = escaping → type-erased.

Цена в checker: нужен **новый, но узкий проход escape-анализа**, а не общий alias-анализ. Данных достаточно — `ExprLambda`, `ExprSpawn`, `StmtReturn`, типы полей уже в AST/registry. Проход: для каждого `ExprLambda` определить, попадает ли его значение в один из 4 escaping-контекстов (простой обход вверх по синтаксическому родителю / по потоку данных внутри функции — не межпроцедурно). Ровно как Swift: один бит на замыкание.

Класс миграции: **breaking для codegen** (меняется форма генерируемого C++ для параметров-функций), но **non-breaking для языка** (существующий MLC-код не меняется, меняется только что генерируется). Это высокая ценность (прямо бьёт в названную боль скорости сборки) при средней стоимости. Порядок внедрения по правилам проекта: сначала Ruby-эталон escape-анализа, потом mlcc.

# Часть 3: приоритизированный список (ценность / стоимость)

**1. Array HOF (A1) — non-breaking + runtime.** Что: `map`/`filter`/`fold`/... на `[T]`. Зачем: приведённый реальный код (`ast.mlc`, `names.mlc`, `registry.mlc`) буквально состоит из ручных `while index < n do ... index = index+1` — это 80% названных 210 циклов; без HOF self-hosted компилятор нельзя писать идиоматично, что и признано критерием "не хуже Ruby". Прямая C++-трансляция в `std::transform`/`copy_if`/`accumulate`, ноль нового рантайма кроме методов на `mlc::Array`. Наивысший ROI, уже приоритет №1 проекта.

**2. Инверсия дефолта замыканий на non-escaping (Часть 1b.3) — breaking codegen, non-breaking язык.** Что: замыкания-параметры по умолчанию → `template<class F>` (ноль RC/heap), escaping выводится узким анализом (4 правила) + явный маркер `escaping Fn(...)`. Зачем: прямо бьёт в названную в задаче первопричину кризиса скорости C++-сборки (*повсеместные `std::function`/`shared_ptr`*). Swift `@escaping` — проверенный образец, что это облегчённый анализ, не Rust-borrow. Высокая ценность, средняя стоимость (новый узкий проход, не alias-анализ).

**3. Result/Option комбинаторы + `?` с `From`-конверсией (A2 + 2.3) — non-breaking sugar.** Что: `and_then`/`map`/`map_err`/`or_else` + расширение `?` авто-конверсией `E` через `From`. Зачем: убирает вложенные `match` (названная боль A2), а `From`-конверсия делает railway-стиль работоспособным на смешанных типах ошибок — без неё цепочки ломаются. Методы на `std::variant`, zero overhead.

**4. `Weak` эргономика: `.weak()`/`.upgrade() -> Option` (Уровень 0, 1.3) — non-breaking sugar.** Что: методы downgrade/upgrade на `Shared`/`Weak`. Зачем: `grep "Weak<"` = 0 — механизм разрыва циклов задокументирован, но непригоден к использованию, значит единственный санкционированный способ управления циклами в языке де-факто мёртв. Дешевле некуда (`std::weak_ptr::lock()`), возвращает `Option` — совместимо с существующим match.

**5. Lint циклических типов без Weak (Уровень 1, 1.3) — non-breaking warning.** Что: Tarjan-SCC по графу типов из `TypeRegistry.adt_index`, warning на цикл `Shared` без `Weak`. Зачем: компенсирует зафиксированное ограничение *"Забытый `Weak` → утечка. Компилятор не детектирует"* — превращает молчаливую утечку в диагностику. Один файл в checker, данные уже есть. Эвристика → warning с suppression, не error.

**6. Or-паттерны с биндингом + guards с корректной exhaustiveness (2.4, B2/E2) — non-breaking.** Что: завершить `Circle(r) | Square(r) => ...` (проверка совпадения имён альтернатив, сейчас `names.mlc` берёт только `alternatives[0]`) + `if`-guards, где guard-ветка не покрывает exhaustiveness. Зачем: pattern matching — центральный инструмент языка над ADT, эти пробелы заставляют дублировать ветки. Чистый сахар/правила checker.

**7. Orphan rule для `extend` (2.5) — non-breaking пока кодовая база мала.** Что: `extend T : Trait` только в модуле объявления `T` или `Trait`. Зачем: сейчас правила когерентности нет; при росте (self-host уже 5200+ строк, много модулей) конфликтующие impl станут трудноуловимым классом ошибок — дешевле ввести правило сейчас, пока нарушений мало. Проверка по модулю объявления в registry.

`mlc::memory::RegionHandle` / pmr; `r.alloc` → `Region<Tag,T>`; E091 escape
(return/closure/assign/field). Track:
