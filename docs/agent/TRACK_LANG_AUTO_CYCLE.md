## Разграничение с уже закрытым `CYCLE_LINT`

`CYCLE_LINT` — **compile-time** структурный анализ типов (граф полей
`Shared<T>`, поиск SCC, warning на этапе компиляции). Не видит рантайм-
значения, может давать false positive (структурный цикл типов без
реального цикла значений) — это осознанный trade-off лёгкого lint.

Nim ORC — **runtime** механизм: периодический trial-deletion трейсинг по
всем refcounted объектам для реального обнаружения и разрыва циклов
значений (не типов), без ложных срабатываний, но с runtime cost (проходы
по графу объектов) и существенной сложностью реализации (нужен полный
обход графа объектов с учётом всех типов, включая generic-контейнеры).

Это **не** следующий шаг после `CYCLE_LINT` — принципиально другой слой
(runtime GC-подобный механизм vs compile-time lint), и явно противоречит
принципу проекта "без GC/рантайма" (`PLAN.md` §8: "GC / borrow checker — не
в roadmap").

## Задача трека

Одна design-сессия: явное решение — won't-do (с обоснованием, чтобы не
поднимался вопрос повторно без новых аргументов) или узкий прототип с
чётко ограниченным scope.
