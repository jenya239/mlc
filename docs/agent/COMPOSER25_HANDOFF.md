# Handoff: работа с composer-2.5 (слабее Sonnet)

Причина документа: заканчивается доступ к Sonnet, дальнейшая работа —
отдельный чат на composer-2.5. Модель заметно слабее по глубине
рассуждений о семантике/типах/precedence. Документ фиксирует (1) минимум
контекста, без которого модель не сможет ориентироваться, и (2) правила
постановки задач, чтобы результат оставался проверяемым, а не
"выглядит правдоподобно".

`AGENTS.md`, `CLAUDE.md`, `.cursor/rules/*` подключаются автоматически в
любом чате этого workspace независимо от модели — их содержимое **не
повторяется** здесь. Этот файл — надстройка специально под слабую модель.

---

## 1. Проект в двух словах

- **MLC** — язык программирования, цель — self-hosted компилятор.
- Два независимых пайплайна на **один** язык:
  - **Bootstrap (Ruby)**, `lib/mlc/` — эталон семантики, полный стек
    (лексер → парсер → SemanticIR → C++ AST DSL → C++). Тесты:
    `bundle exec rake test_mlc`.
  - **Self-hosted (mlcc)**, `compiler/` — компилятор, написанный на самом
    MLC, собирается через Ruby (`compiler/build.sh`). Тесты:
    `bundle exec rake test_compiler_mlc`.
- Цель — self-hosted `mlcc` наблюдаемо ведёт себя так же, как Ruby-эталон
  (не архитектурно идентично — по результату компиляции и исполнения).
- Оба пайплайна порождают C++20, который линкуется с `runtime/` (COW
  `Array`/`HashMap`/`String`, `shared_ptr`-обёртки).

## 4. Правила постановки задач для composer-2.5

### 4.1 Формат задачи (обязательно)

Каждая задача, которую даём composer-2.5, должна содержать:

1. **Ровно один файл или одну функцию** как область изменения (максимум
   2 файла, если один из них — codegen-парный для того же конструкта).
2. **Точную ссылку** на текущий код (файл:строка), полученную через
   `Grep`/`Read` заранее — не "найди сам, где это реализовано". Слабая
   модель хуже справляется с открытой навигацией по большому дереву.
3. **Готовый репро** (минимальный `.mlc`-сниппет) с ожидаемым результатом
   "до" и "после" — не "проверь, работает ли". Пример — любой шаг в
   `docs/archive/tracks/TRACK_PARSER_GAPS_H1H4.md`.
4. **Явную команду verify** для копипаста, не описание словами:
   ```bash
   bundle exec rake test_compiler_mlc
   compiler/build.sh
   compiler/out/mlcc /tmp/repro.mlc -o /tmp/repro_out
   ```
5. **Явный запрет** на смежные изменения ("не трогай лексер, если задача
   про парсер", "не трогай `lib/mlc/` в этой задаче").

### 4.2 Хорошие задачи (примеры)

- Добавить один уровень precedence в существующую лестницу парсера по
  образцу соседнего уровня (STEP=1 в TRACK_PARSER_GAPS_H1H4).
- Добавить элемент в список/match по образцу уже существующих элементов
  (STEP=2, STEP=3 там же).
- Добавить тест-кейс в существующий тестовый файл по образцу соседних
  тестов в том же файле.
- Механическая замена паттерна в N файлах, где паттерн задан явным
  примером "было / стало" (как чистка `hardcoded g++` в этой сессии — но
  тот трек уже закрыт, пример для формата задачи).
- Правки документации: обновить таблицу статусов, дописать секцию по
  готовому шаблону соседних секций.

### 4.3 Плохие задачи (не давать без дополнительного дробления)

- Что-либо, требующее **одновременно** менять `checker/` и `codegen/`, —
  максимум для одной задачи. Дробить на checker-шаг и codegen-шаг.
- "Improve X", "оптимизируй Y", "разберись почему Z" без конкретного
  файла/строки/репро — сначала диагностика (см. §4.5), потом отдельная
  задача на фикс.
- Любая задача, где корректность зависит от **precedence операторов**,
  **generic-инстанциации/CTAD**, **std::visit-маршрутизации по вариантам**
  без явного заданного алгоритма — это ровно тот класс багов, что уже
  ловился в проекте только через реальную компиляцию + self-host diff, а
  не "выглядит правильно" (см. `docs/mlc2.md` — CTAD для `Ok`/`Err`,
  precedence-баг принтера, `std::visit` на `bool`, все — `TRACK_BOOTSTRAP_LINK`,
  все стоили нескольких раундов правок даже для сильной модели). Слабой
  модели такое давать только с уже готовым точным алгоритмом-рецептом.
- STEP=4 из TRACK_PARSER_GAPS_H1H4 (постфиксный if) — сложнее остальных
  трёх шагов там же (меняет структуру разбора выражения, не просто
  добавляет уровень) — если после чтения кода задача окажется больше
  одного файла/функции, не пытаться протащить через composer-2.5,
  эскалировать (см. §4.6).

### 4.4 Обязательная дисциплина verify (не доверять "похоже, работает")

- **Никогда** не принимать результат без реального запуска команд из
  задачи и вставки настоящего вывода в ответ. Слабая модель чаще
  утверждает успех без проверки или неверно интерпретирует вывод ошибки.
- Просить явно: "запусти команду, вставь вывод целиком, не резюмируй".
- Полная лестница перед закрытием шага/трека — `docs/agent/DEVELOPMENT.md`
  §Verification ladder, Tier B при коммите: self-host diff обязателен
  при правках `compiler/**`.
- Если self-host diff (`mlcc` vs `mlcc2`) не совпал — это блокер, не
  "мелкая деталь". Останавливаться, не продолжать следующий шаг.

### 4.5 Если задача "не лезет" в одну атомарную формулировку

1. Сначала — отдельная задача **только на диагностику**: "прочитай файлы
   X, Y; найди, где именно обрабатывается Z; процитируй конкретные
   строки" — без правок кода.
2. По результату диагностики — новая, уже узкая задача на правку.
3. Не позволять модели сама решать архитектуру/scope многошаговой задачи
   — дробление делает оператор (человек или более сильная модель в
   другом чате), не composer-2.5 сама себе.

### 4.6 Эскалация на API (Opus/Sonnet) вместо composer-2.5

Для задач, требующих **чтения** большого объёма кода одновременно для
консистентности (не для генерации кода с нуля) — не пытаться пропустить
через composer-2.5 частями, а использовать `mlc-support/` (соседний
репозиторий, Ruby-скрипты, прямые запросы к Anthropic API):

- `bundle_sources.sh` / `bundle_parity_sources.sh` — собрать бандл исходников.
- `implement_step_generic.rb --dry-run` — **всегда сначала dry-run**,
  проверить токены/стоимость, затем реальный запрос.
- `review_mlc2_parity.rb`, `review_with_claude.rb` — read-only отчёт/анализ,
  не правки кода напрямую.
- `apply_response.rb --dry-run` — применить SEARCH/REPLACE из ответа,
  сначала проверить без записи.
- Полный список кандидатов и обоснование, когда большой контекст
  оправдан: `docs/LARGE_CONTEXT_TASKS.md`.
- **Результат API-запроса — тоже не готовая правка.** Через него проходит
  тот же verify gate, что и через ручную правку (см. §4.4). Пример
  дисциплины — как в этой сессии: 12 сырых находок от Opus, 4 вручную
  перепроверены на бинаре и только они пошли в `docs/mlc2.md`, остальные
  8 остались пометкой "гипотеза, не проверено" в `docs/PARITY_AUDIT_2026_07.md`.
- Любой запрос к API — **сначала dry-run с оценкой стоимости**, показать
  оператору, дождаться подтверждения перед реальной (платной) отправкой.
  `.env` с `ANTHROPIC_API_KEY` уже настроен в `mlc-support/`.

### 4.7 Коммиты

- Без изменений — правило из `CLAUDE.md`/user rules не меняется:
  **коммит только по явной команде**, без `--trailer`, без
  co-authored-by, тон сообщения — как в истории репозитория (короткий
  императив на английском, `git log --oneline -20` для образца).
- Один коммит — одна задача/один concern. Не смешивать несколько шагов
  трека в одном коммите даже если оператор забыл напомнить.

## 5. Рекомендуемая первая задача

`docs/archive/tracks/TRACK_PARSER_GAPS_H1H4.md`, **STEP=1** (побитовые операторы) —
самый механический из четырёх, полный репро и точные ссылки на строки уже

см. §4.3; если после чтения кода не укладывается в один файл/функцию —
эскалировать по §4.6, не пытаться протолкнуть).

Формулировка задачи оператору для составления промпта composer-2.5 —
взять целиком секцию `## STEP=1` из трека, добавить в конец: «выполни
только это, запусти verify-команды из шапки трека, вставь вывод целиком».
