# Инструкция: как составлять план реализации фичи MLC

Для агента и человека: порядок обсуждения и структура плана так же, как для E1/B1 (derive), D4 и подобных задач.

## Привязка к документу

- Найти в `docs/MLC.md` **конкретную секцию** (E1, B1, D4 и т.д.). Не смешивать номера и формулировки разных секций в один статус «сделано».
- Если в документе противоречия между секциями (разный обещанный C++, строки «уже реализовано» не совпадают с кодом), в плане явно записать **расхождение** и пункт выравнивания документа или кода.

## Проверка факта в коде

- Перед формулировкой «реализовано» сделать проверку через поиск по репозиторию и чтение ключевых файлов (пути из `CLAUDE.md`: Ruby semantic/codegen в `lib/mlc/`, mlcc в `compiler/`).
- Для фич, которые должны быть в **обоих** компиляторах, статус закрыт только если проверены **Ruby и mlcc**.

## Новые ключевые слова

- В плане отдельный подпункт: нужно ли **новое ключевое слово** или достаточно расширить существующую грамматику (например новое имя в `derive { … }`).
- Если ключевые слова не добавляются — написать это явно.

## Контракт до кодирования (этап 0)

- Таблица: пользовательский ввод → результат (IR, C++, или ошибка checker).
- Для codegen согласовать с runtime (`runtime/include/mlc/`), стилем генерируемого C++, границей MVP (например только типы без параметров).
- Если возможны варианты (например `operator<` против `operator<=>`), выбрать **один** и записать в план до реализации.

## Порядок работ

Следовать `docs/MLC.md`, раздел про процесс: эталон **Ruby** и `test/mlc/`, затем **mlcc** и `compiler/tests/`, затем `compiler/build.sh` и при необходимости bootstrap (`MLCC_BOOTSTRAP=1`, опционально `MLC_REQUIRE_BOOTSTRAP_LINK=1`).

## Тестирование в плане

Включать развёрнуто, не одной строкой:

- Уровни: lowering в текст C++ (Ruby и mlcc), ошибки checker, **паритет** одних и тех же сниппетов на двух компиляторах, при необходимости компиляция C++ или e2e, регресс `rake test_mlc` и `rake test_compiler_mlc`.
- Матрица частных случаев: синтаксис, граничные типы (пустая запись, sum types), комбинации опций, generics и явная граница MVP, конфликты с пользовательским кодом, негативные сценарии.
- Указать, когда достаточно unit-тестов mlcc, а когда обязателен полный bootstrap.

## Риски и нецели

- Что сознательно **не** входит в первый pull request.
- Риски и смягчение: дублирование с ручным кодом, расхождение Ruby/mlcc, ODR.

## Где хранить план

- Файл в `.cursor/plans/` с кратким именем и при необходимости YAML frontmatter (`overview`, список `todos`).

## Антипаттерны

- Утверждать готовность по памяти без проверки кода.
- Сливать статус двух разных пунктов дорожной карты (например B1 и E1) без таблицы соответствия документ ↔ код.
- План из двух пунктов «сделать» и «протестировать» без контракта и без матрицы случаев.
- Забыть паритет Ruby/mlcc для фич self-hosted цепочки.
