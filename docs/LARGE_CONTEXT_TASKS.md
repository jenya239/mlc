# Кандидаты на batch-запросы к модели с большим контекстом

Оценка от 2026-07-03: какие части проекта имеет смысл прогонять большим
одноразовым (или малосерийным) запросом к внешней модели (инструменты —
`mlc-support/`), а какие требуют итеративного TDD-цикла с реальной
компиляцией. См. также `mlc-support/README.md` (механика запросов,
кэширование, стоимость).

## Ключевое ограничение

Большой контекст помогает **читать** (весь `compiler/` — 48.8k строк MLC,
весь `lib/mlc/` — 32.5k строк Ruby, `runtime/` — 30.5k строк C++ — всё это
влезает в контекст одновременно), но **не помогает писать** — лимит вывода
на порядок меньше миллиона токенов. «Переписать весь self-hosted компилятор
одним запросом» невозможно вне зависимости от размера контекста — вывод
всё равно придётся резать на много запросов.

Также не факт, что кандидату вообще нужен большой контекст в смысле
объёма — иногда нужно просто «увидеть несколько файлов разом для
консистентности», а токенов там на порядки меньше миллиона (см. пример
пункта 1 ниже — 19.5K токенов, не 1M).

## Где НЕ имеет смысла

`compiler/checker/`, `compiler/codegen/`, `compiler/mir/` — ядро семантики и
кодогена C++. Баги здесь не ловятся "выглядит правдоподобно": precedence-баг
принтера (`(A||B)&&C` → `A||B&&C`), CTAD для `Ok`/`Err`, unsafe-тернарники
между альтернативами суммовых типов, неверная маршрутизация `std::visit` на
`bool` (все — `TRACK_BOOTSTRAP_LINK`, закрыт 2026-07-03) — все компилировались,
выглядели корректно на глаз, ловились только через реальный компилятор +
`test_compiler_mlc` + `regression_gate.sh` + self-hosting identity diff на
каждой итерации. Большой контекст даёт больше правдоподобного текста, не
больше корректности. Одноразовая генерация такого объёма без цикла
компиляция→ошибка→правка почти гарантированно тихо проглотит похожие баги.

## Про "сгенерировать с нуля"

Не для существующего self-hosted компилятора целиком — Ruby-эталон
(`lib/mlc/`) содержит годы накопленных edge-case фиксов, пересоздание с нуля
выбрасывает это знание. Имеет смысл только для ещё не начатых модулей без
существующего self-hosted кода (пункт 3 выше — там "с нуля" и "порт большим
контекстом" совпадают).

## Инструменты

`mlc-support/` (соседний репозиторий) — Ruby-скрипты для запросов к
Anthropic API с кэшированием контекста:
- `bundle_sources.sh` — полный бандл `compiler/*.mlc`.
- `implement_step_slim.rb` — для задач, где все файлы `.mlc` (авто-извлечение
  путей из текста шага по regex).
- `implement_step_generic.rb` — то же для задач вне MLC (bash/YAML/markdown),
  без инструкции "только синтаксис MLC".
- Оба поддерживают `--dry-run` (точный подсчёт токенов через бесплатный
  `count_tokens`-эндпоинт, без реальной дорогой генерации).
- `apply_response.rb` — применяет SEARCH/REPLACE блоки из ответа к репозиторию
  `mlc` (`--dry-run` для проверки без записи).
- `steps/` — сохранённые описания шагов (переиспользуемая история задач).

Независимо от способа генерации — результат обязан пройти тот же gate
(`test_compiler_mlc`, `regression_gate.sh`, self-hosting identity diff), что
и обычные правки.
